## Часть 1: Подготовка лабораторного стенда

### Задание 1.1: Создание схемы и тестовых данных

Создана отдельная схема `injection_lab` с упрощёнными таблицами `users` (3 строки) и `tasks` (4 строки).

```sql
CREATE SCHEMA IF NOT EXISTS injection_lab;

CREATE TABLE injection_lab.users (
    user_id SERIAL PRIMARY KEY,
    username TEXT NOT NULL UNIQUE,
    full_name TEXT NOT NULL,
    role_name TEXT NOT NULL,
    email TEXT NOT NULL UNIQUE,
    password_plain TEXT NOT NULL
);

CREATE TABLE injection_lab.tasks (
    task_id SERIAL PRIMARY KEY,
    title TEXT NOT NULL,
    description TEXT,
    status TEXT NOT NULL DEFAULT 'new',
    priority TEXT NOT NULL DEFAULT 'medium',
    assignee_username TEXT NOT NULL REFERENCES injection_lab.users(username),
    created_at TIMESTAMP NOT NULL DEFAULT NOW()
);
```

**Проверка:** `SELECT COUNT(*) FROM injection_lab.users` → 3, `SELECT COUNT(*) FROM injection_lab.tasks` → 4.

## Часть 2: Анализ уязвимых фрагментов кода

### Задание 2.1: Уязвимая аутентификация

**Уязвимый код:**
```javascript
async function login(client, username, password) {
  const sql =
    "SELECT user_id, username, role_name " +
    "FROM injection_lab.users " +
    "WHERE username = '" + username + "' " +
    "AND password_plain = '" + password + "'";

  return client.query(sql);
}
```

**Почему код уязвим:**
Пользовательский ввод (`username`, `password`) подставляется в SQL-запрос через конкатенацию строк без экранировки или параметризации. Злоумышленник может внедрить произвольный SQL-код через поля ввода.

**Пример вредоносного ввода для обхода аутентификации:**
- `username`: `' OR 1=1 --`
- `password`: `anything` (любое значение)

**Итоговый SQL-запрос после подстановки:**
```sql
SELECT user_id, username, role_name
FROM injection_lab.users
WHERE username = '' OR 1=1 --' AND password_plain = 'anything'
```

**Разбор:**
1. `username = ''` — ложно для всех пользователей
2. `OR 1=1` — всегда истинно → всё условие WHERE истинно
3. `--` — комментирует остаток запроса (проверку пароля)
4. **Результат:** возвращаются **ВСЕ 3 пользователя** — обход аутентификации

**Данные, которые может получить злоумышленник:**
- `user_id`, `username`, `role_name` всех пользователей
- Приложение может авторизовать атакующего как первого пользователя в результате (возможно, администратора)
- Через UNION-инъекцию можно извлечь email и пароли

### Задание 2.2: Уязвимый поиск задач

**Уязвимый код:**
```javascript
async function findTasksByStatus(client, status) {
  const sql =
    "SELECT task_id, title, status, priority " +
    "FROM injection_lab.tasks " +
    "WHERE status = '" + status + "'";

  return client.query(sql);
}
```

**Пример инъекции, возвращающей все задачи:**
- `status`: `' OR 1=1 --`

**Итоговый запрос:**
```sql
SELECT task_id, title, status, priority
FROM injection_lab.tasks
WHERE status = '' OR 1=1 --'
```

**Как работает комментарий `--`:**
Двойной дефис `--` в SQL означает начало однострочного комментария. Всё, что стоит после `--`, PostgreSQL игнорирует. Это позволяет «отбросить» закрывающую кавычку, которую добавляет код приложения.

**Сравнение результатов:**

| Запрос | Ввод | Результат |
|--------|------|-----------|
| Нормальный | `status = 'new'` | 2 задачи (только со статусом `new`) |
| Атакованный | `status = '' OR 1=1 --` | **4 задачи** (все, включая `in_progress` и `done`) |

### Задание 2.3: Уязвимая сортировка

**Уязвимый код:**
```javascript
async function listTasks(client, sortField, sortDirection) {
  const sql =
    "SELECT task_id, title, priority, created_at " +
    "FROM injection_lab.tasks " +
    "ORDER BY " + sortField + " " + sortDirection;

  return client.query(sql);
}
```

**Почему параметризация неприменима напрямую к имени столбца:**
Параметры (`$1`, `$2`) в PostgreSQL подставляются как **значения** (литералы), а не как **идентификаторы**. Имя столбца — это идентификатор SQL. Если написать `ORDER BY $1`, PostgreSQL интерпретирует `$1` как строковое значение, а не как имя колонки — запрос вернёт ошибку или будет сортировать по константе.

**Риски использования `sortField` и `sortDirection` без проверки:**
1. **Утечка данных через подзапрос:** `sortField = "(SELECT password_plain FROM injection_lab.users LIMIT 1)"` — извлечение паролей через ORDER BY
2. **Деструктивные действия:** `sortField = "title; DROP TABLE injection_lab.tasks; --"` — удаление таблицы
3. **Повышение привилегий:** `sortField = "title; UPDATE injection_lab.users SET role_name='admin' WHERE username='alice'; --"`
4. **Разведка структуры БД:** `sortField = "(SELECT column_name FROM information_schema.columns LIMIT 1)"`

**Безопасная стратегия:** Использование **белого списка** (whitelist) — словаря разрешённых значений с подстановкой значения по умолчанию при несовпадении.

## Часть 3: Исправление уязвимостей

### Задание 3.1: Безопасная аутентификация (параметризация)

**Исправленный код:**
```javascript
async function loginSafe(client, username, password) {
  const sql = {
    text:
      "SELECT user_id, username, role_name " +
      "FROM injection_lab.users " +
      "WHERE username = $1 AND password_plain = $2",
    values: [username, password]
  };

  return client.query(sql);
}
```

**Почему `' OR 1=1 --` больше не изменит логику запроса:**
При параметризации значение `$1` воспринимается PostgreSQL как **литерал**. Строка `' OR 1=1 --` не интерпретируется как SQL-код, а передаётся целиком как значение для сравнения. PostgreSQL ищет пользователя с username, равным буквально `' OR 1=1 --` — такого нет, результат пустой.

**Проверка через PREPARE/EXECUTE:**
```sql
PREPARE login_safe(TEXT, TEXT) AS
    SELECT user_id, username, role_name
    FROM injection_lab.users
    WHERE username = $1 AND password_plain = $2;

-- Корректный ввод:
EXECUTE login_safe('alice', 'alice123');
-- Результат: 1 строка (alice, employee)

-- Вредоносный ввод:
EXECUTE login_safe(''' OR 1=1 --', 'anything');
-- Результат: 0 строк (инъекция НЕ сработала)
```

### Задание 3.3: Безопасная сортировка (белые списки)

**Исправленный код:**
```javascript
async function listTasksSafe(client, sortField, sortDirection) {
  const allowedFields = {
    created_at: "created_at",
    priority:   "priority",
    title:      "title",
    status:     "status"
  };

  const allowedDirections = {
    asc:  "ASC",
    desc: "DESC"
  };

  const field = allowedFields[sortField] ?? "created_at";
  const direction = allowedDirections[sortDirection] ?? "DESC";

  const sql =
    "SELECT task_id, title, priority, created_at, status " +
    "FROM injection_lab.tasks " +
    "ORDER BY " + field + " " + direction;

  return client.query(sql);
}
```

**Проверка для допустимых значений:**
- `sortField = 'title'`, `sortDirection = 'asc'` → `ORDER BY title ASC` — корректная сортировка
- `sortField = 'priority'`, `sortDirection = 'desc'` → `ORDER BY priority DESC` — корректная сортировка

**Проверка для недопустимого значения:**
- `sortField = 'title; DROP TABLE injection_lab.tasks; --'`
- `allowedFields['title; DROP TABLE injection_lab.tasks; --']` → `undefined`
- `field = "created_at"` (значение по умолчанию)
- Итоговый запрос: `ORDER BY created_at DESC` — **вредоносная строка полностью проигнорирована**

## Часть 4: Проверка защиты на уровне БД

### Задание 4.1: Ограниченная роль приложения

```sql
CREATE ROLE web_app_demo LOGIN PASSWORD 'demo123';

REVOKE ALL ON SCHEMA injection_lab FROM PUBLIC;
GRANT USAGE ON SCHEMA injection_lab TO web_app_demo;

REVOKE ALL ON ALL TABLES IN SCHEMA injection_lab FROM web_app_demo;
GRANT SELECT ON injection_lab.users TO web_app_demo;
GRANT SELECT ON injection_lab.tasks TO web_app_demo;
GRANT UPDATE(status) ON injection_lab.tasks TO web_app_demo;
```

**Как это ограничивает ущерб при уязвимости:**
- `DROP TABLE` — невозможен (нет прав DDL)
- `DELETE FROM` — невозможен (нет права DELETE)
- `INSERT` — невозможен (нет права INSERT)
- `UPDATE` — ограничен **только полем `status`** в таблице `tasks`
- Даже при успешной инъекции злоумышленник не сможет модифицировать или уничтожить данные

### Задание 4.2: Матрица прав

| Параметр | web_app_demo | Владелец схемы | Суперпользователь |
|----------|-------------|----------------|-------------------|
| SELECT | users, tasks | ВСЕ таблицы | ВСЕ + системные |
| INSERT | — | ВСЕ таблицы | ВСЕ |
| UPDATE | tasks.status | ВСЕ поля | ВСЕ |
| DELETE | — | ВСЕ таблицы | ВСЕ |
| DROP/ALTER TABLE | — | ДА | ДА |
| pg_shadow | — | — | ДА |
| CREATE EXTENSION | — | — | ДА |
| COPY TO/FROM | — | — | ДА |

**Вывод:** роль с минимальными привилегиями существенно снижает потенциальный ущерб от SQL-инъекции.

## Выводы

1. **SQL-инъекции** остаются одной из наиболее опасных уязвимостей, даже при наличии правильно настроенных ролей и RLS.
2. **Параметризованные запросы** ($1, $2) — основной метод защиты для пользовательских **значений** (WHERE, INSERT, UPDATE).
3. **Белые списки** — необходимый метод для пользовательских **идентификаторов** (ORDER BY, имена таблиц/столбцов), так как параметризация к ним неприменима.
4. **Принцип наименьших привилегий** — дополнительный уровень защиты: роль `web_app_demo` ограничивает последствия даже при успешной атаке.
5. **Многоуровневая защита** (параметризация + белые списки + минимальные права + RLS) обеспечивает наибольшую безопасность.
